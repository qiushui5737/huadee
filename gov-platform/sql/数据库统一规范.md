# 政府网站集约化平台数据库统一规范

本文档用于统一五位负责人在开发过程中的数据库使用方式，避免出现表结构不一致、字段命名不一致、脚本无法重复执行等问题。

## 1. 统一库

项目统一使用单库模式：

```text
数据库名：gov_platform
字符集：utf8mb4
排序规则：utf8mb4_0900_ai_ci
```

后端统一连接配置：

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/gov_platform?useUnicode=true&characterEncoding=utf-8&serverTimezone=Asia/Shanghai
    username: root
    password: 050420
```

本地首次初始化执行：

```powershell
& 'C:\Program Files\MySQL\MySQL Server 8.0\bin\mysql.exe' -uroot -p050420 --default-character-set=utf8mb4 -e "source H:/huadishixixiangmu/huadee/gov-platform/sql/init_gov_platform_full.sql"
```

说明：

- 所有人本地数据库都叫 `gov_platform`。
- 所有人都从 `sql/init_gov_platform_full.sql` 初始化。
- 不直接修改已经在线上或其他同学电脑执行过的建表语句；后续变更写到 `sql/migrations/`。

## 3. 公共表使用规则

公共表由 E 负责维护，其他负责人可以读取或通过接口使用，但不要私自改字段。

| 公共表 | 使用场景 |
| --- | --- |
| `sys_user` | 审批人、发布人、回复人、审核人 |
| `sys_dept` | 部门子站、留言分派、办事事项所属部门 |
| `sys_role` / `sys_permission` | 管理端菜单和按钮权限 |
| `sys_message` | 办件通知、留言回复通知、审核提醒 |
| `performance_metric` | 部门绩效评分、排名和报表 |

跨模块调用建议通过后端接口完成，不建议直接跨模块随意写对方负责的业务表。

## 4. 统一字段规范

### 命名规范

| 类型 | 规范 | 示例 |
| --- | --- | --- |
| 表名 | 小写下划线 | `service_record` |
| 字段名 | 小写下划线 | `create_time` |
| 主键 | 统一使用 `id` | `id BIGINT PRIMARY KEY AUTO_INCREMENT` |
| 编码字段 | 使用 `xxx_code` | `dept_code`, `item_code` |
| 名称字段 | 使用 `xxx_name` | `dept_name`, `item_name` |
| 状态字段 | 使用 `status` | `pending`, `published` |
| 用户字段 | 使用 `user_id` | `user_id BIGINT` |
| 部门字段 | 使用 `dept_code` | `dept_code VARCHAR(50)` |

### 通用字段

业务主表建议包含：

```sql
id BIGINT PRIMARY KEY AUTO_INCREMENT,
create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
deleted TINYINT DEFAULT 0
```

只做记录流水的表可以不加 `update_time` 和 `deleted`，例如审核记录、日志、消息发送记录。

### 状态值规范

状态字段统一使用英文小写，避免中文状态导致编码或比较问题。

常用状态：

| 场景 | 推荐值 |
| --- | --- |
| 通用启停 | `enabled`, `disabled` 或 `1`, `0` |
| 审核 | `pending_review`, `approved`, `rejected`, `published` |
| 办件 | `accepted`, `processing`, `approved`, `rejected`, `finished` |
| 留言 | `pending_assign`, `assigned`, `replied`, `closed` |
| 支付 | `unpaid`, `paid`, `refunded`, `closed` |
| 任务 | `pending`, `running`, `success`, `failed` |

## 5. 统一脚本管理

### 目录结构

```text
sql/
  init_gov_platform_full.sql      # 首次初始化脚本
  数据库统一规范.md                # 数据库协作规范
  migrations/                     # 后续变更脚本
    README.md
    VYYYYMMDD_序号_简短说明.sql
```

### 初始化脚本

`init_gov_platform_full.sql` 只用于首次建库，要求：

- 使用 `CREATE DATABASE IF NOT EXISTS`
- 使用 `CREATE TABLE IF NOT EXISTS`
- 初始化数据使用 `INSERT IGNORE`
- 不写 `DROP DATABASE`
- 不写 `DROP TABLE`

### 迁移脚本

后续任何表结构变化都写到 `sql/migrations/`，命名格式：

```text
V日期_序号_说明.sql
```

示例：

```text
V20260716_01_add_service_record_handle_dept.sql
V20260716_02_add_message_priority.sql
V20260716_03_create_survey_statistics.sql
```

迁移脚本要求：

- 每个脚本只做一个清晰变更。
- 先写注释说明负责人、原因、影响模块。
- 新增字段尽量允许为空或给默认值，避免破坏已有数据。
- 修改公共表前必须通知全组。
- 禁止随意删除字段、删除表、清空数据。

## 6. 五人协作流程

1. 每个人先执行统一初始化脚本，保证本地库结构一致。
2. 每个人只维护自己模块的业务表。
3. 如果需要改公共表，先和 E 或组长确认。
4. 新增字段或新表时，新增 migration 脚本。
5. 提交代码时同时提交对应 SQL 迁移脚本。
6. 每次拉取别人代码后，先检查 `sql/migrations/` 是否有新脚本。
7. 启动后端前确认 `application.yml` 指向 `gov_platform`。

## 7. 推荐执行检查

查看数据库是否存在：

```sql
SHOW DATABASES LIKE 'gov_platform';
```

查看表数量：

```sql
SELECT COUNT(*) AS table_count
FROM information_schema.tables
WHERE table_schema = 'gov_platform';
```

查看模块关键表：

```sql
SHOW TABLES FROM gov_platform;
```

查看基础数据：

```sql
SELECT COUNT(*) FROM gov_platform.sys_user;
SELECT COUNT(*) FROM gov_platform.service_item;
SELECT COUNT(*) FROM gov_platform.cms_content;
```
