## 1. 项目概述

### 1.1 项目背景

当前各级政府部门网站分散建设、独立运维，存在重复投资、信息孤岛、标准不一、群众办事不便等突出问题。为贯彻落实国务院关于"加快推进全国一体化政务服务平台建设"的指导意见，本项目旨在建设统一的政府网站集约化平台，将分散的部门网站整合到统一平台，实现"一站式"政务服务，提升政府治理能力和公共服务水平。

### 1.2 项目目标

| 序号 | 目标 | 描述 |
|------|------|------|
| 1 | 集约化建设 | 整合教育部、社会保障部、财政部等部门网站，实现统一基础设施、统一标准规范、统一安全防护 |
| 2 | 一站式服务 | 群众可通过统一入口办理各类政务事项，无需在多个部门网站间跳转 |
| 3 | 智能化服务 | 引入Spring AI智能问答、智能搜索等AI能力，提升服务效率和用户体验 |
| 4 | 数据互通 | 打通群众端与管理端数据链路，实现"一网通办、一网统管" |
| 5 | 阳光政务 | 通过依申请公开、政民互动等功能，保障公众知情权、参与权和监督权 |

### 1.3 项目范围

本项目涵盖以下两大端、七大子系统：

```
政府网站集约化平台
├── 群众客户端（C端）
│   ├── 智能搜索子系统
│   ├── 智能问答子系统（Spring AI）
│   ├── 办事服务子系统
│   ├── 政民互动子系统
│   ├── 部门集约化子系统
│   └── 在线依申请公开管理子系统
│
└── 政府管理端（B端）
    ├── 统计分析子系统
    ├── 政民互动管理子系统
    └── 服务绩效管理子系统
```

### 1.4 术语与缩略语

| 术语 | 说明 |
|------|------|
| 集约化 | 将分散建设的网站整合到统一平台，共享基础设施和技术资源 |
| C端 | 群众客户端，面向公众用户 |
| B端 | 政府管理端，面向政府工作人员 |
| 依申请公开 | 公民、法人可依法向政府申请获取尚未主动公开的政府信息 |
| SSO | Single Sign-On，单点登录 |
| 流式回答 | AI问答采用SSE（Server-Sent Events）流式输出，逐字返回 |

## 2. 总体架构需求

### 2.1 架构设计原则

1. **前后端分离**：前端采用Vue3/React，后端采用Spring Boot 3 + Spring Cloud，通过RESTful API通信
2. **微服务化**：各子系统按业务域拆分为独立微服务，独立部署、独立扩展
3. **集约共享**：基础设施共享（统一认证、统一消息、统一搜索、统一工作流）
4. **安全合规**：符合国家网络安全等级保护2.0三级要求，支持国产化适配
5. **AI驱动**：核心交互场景引入Spring AI，提升智能化水平

### 2.2 技术栈选型

| 层次 | 技术选型 | 说明 |
|------|----------|------|
| 前端框架 | Vue 3 + TypeScript + Element Plus | C端响应式适配PC/移动端；B端管理后台 |
| 移动端 | UniApp / 微信小程序 | 群众端移动适配 |
| 后端框架 | Spring Boot 3.2 + Spring Cloud 2023 | Java 17+ |
| AI框架 | Spring AI 1.0 | 智能问答、语义搜索 |
| 微服务网关 | Spring Cloud Gateway | 统一路由、限流、鉴权 |
| 注册中心 | Nacos 2.x | 服务注册与配置中心 |
| 搜索引擎 | Elasticsearch 8.x | 智能搜索、全文检索 |
| 关系数据库 | MySQL 8.0 / 达梦数据库 | 业务数据存储 |
| 缓存 | Redis 7.x | 会话缓存、热点数据、限流 |
| 消息队列 | RabbitMQ | 异步消息、事件通知 |
| 流式输出 | SSE (Server-Sent Events) | AI问答流式响应 |
| 文件存储 | MinIO | 附件、材料文件存储 |
| 工作流引擎 | Flowable | 办事审批、依申请公开流程 |
| 对象关系映射 | MyBatis-Plus | 数据持久层 |
| 监控 | Prometheus + Grafana | 系统监控 |
| 日志 | ELK (Elasticsearch + Logstash + Kibana) | 日志收集分析 |
| 容器化 | Docker + Kubernetes | 部署编排 |

## 3. 群众客户端需求分析

### 3.1 智能搜索子系统

#### 3.1.1 功能概述

为群众提供全站统一搜索入口，基于Elasticsearch实现全文检索，支持关键词联想、智能纠错、分类筛选和结果排序。

#### 3.1.2 详细功能需求

| 编号 | 功能项 | 描述 | 优先级 |
|------|--------|------|--------|
| SR-01 | 关键词搜索 | 用户输入关键词，搜索平台所有公开内容（新闻、政策、办事指南、公告等） | P0 |
| SR-02 | 搜索联想 | 输入时实时推荐相关搜索词（下拉联想） | P1 |
| SR-03 | 智能纠错 | 对拼写错误的关键词进行自动纠错提示 | P2 |
| SR-04 | 分类筛选 | 按内容类型（政策文件、办事指南、新闻动态、公告公示）筛选搜索结果 | P0 |
| SR-05 | 按部门筛选 | 按发布部门（教育部、社会保障部、财政部等）筛选 | P1 |
| SR-06 | 按时间筛选 | 按发布时间范围（今日/本周/本月/自定义）筛选 | P1 |
| SR-07 | 结果排序 | 支持按相关度、发布时间排序 | P0 |
| SR-08 | 搜索热词 | 展示当前热搜词排行榜 | P2 |
| SR-09 | 搜索历史 | 记录用户搜索历史（需登录） | P2 |
| SR-10 | 语义搜索 | 基于Spring AI Embedding向量检索，理解用户意图而非仅匹配关键词 | P1 |
| SR-11 | 搜索结果高亮 | 关键词在搜索结果中高亮显示 | P1 |
| SR-12 | 空结果推荐 | 无搜索结果时推荐热门相关内容 | P2 |

#### 3.1.3 数据索引设计

搜索索引需覆盖以下数据源：

| 数据源 | 索引字段 | 更新方式 |
|--------|----------|----------|
| 政策文件库 | 标题、文号、正文、发布部门、发布日期、主题分类 | 增量同步（MQ通知） |
| 办事指南库 | 事项名称、办理流程、申请材料、办理部门、适用范围 | 增量同步 |
| 新闻动态库 | 标题、正文、发布部门、发布时间 | 增量同步 |
| 公告公示库 | 标题、正文、发布部门、有效期限 | 增量同步 |
| 依申请公开目录 | 信息名称、内容描述、产生日期、公开属性 | 定时全量+增量 |

#### 3.1.4 接口需求

```
GET  /api/search/suggest?keyword=xxx          // 搜索联想
GET  /api/search?keyword=xxx&type=&dept=&startDate=&endDate=&page=&size=&sort=   // 搜索
GET  /api/search/hot-words                    // 热搜词
GET  /api/search/history                      // 搜索历史（需认证）
```

### 3.2 智能问答子系统

#### 3.2.1 功能概述

基于Spring AI接入大语言模型，为群众提供7×24小时智能政务问答服务。支持多轮对话记忆、流式输出回答、敏感词过滤，覆盖政策咨询、办事指南、常见问题等场景。

#### 3.2.2 详细功能需求

| 编号 | 功能项 | 描述 | 优先级 |
|------|--------|------|--------|
| QA-01 | 流式问答 | 采用SSE流式输出，逐字返回AI回答，提升用户体验 | P0 |
| QA-02 | 多轮对话记忆 | 支持上下文记忆，能理解基于前序问答的追问 | P0 |
| QA-03 | 敏感词过滤 | 输入端过滤敏感词（政治敏感、暴力、违法等），输出端二次过滤 | P0 |
| QA-04 | 政策知识库问答 | 基于政务知识库RAG（检索增强生成），回答政策相关问题 | P0 |
| QA-05 | 意图识别 | 识别用户意图（政策咨询/办事指引/投诉建议/闲聊），路由到不同处理逻辑 | P1 |
| QA-06 | 推荐追问 | 回答结束后推荐2-3个相关问题供用户继续追问 | P1 |
| QA-07 | 会话管理 | 支持保存历史会话列表，用户可查看历史问答记录 | P1 |
| QA-08 | 人工转接 | AI无法回答时支持转接人工客服（政民互动留言） | P2 |
| QA-09 | 多端支持 | PC端、移动端均支持问答交互 | P0 |
| QA-10 | 回答溯源 | AI回答标注引用的政策文件来源 | P2 |
| QA-11 | 快捷问题 | 首页展示常见快捷问题入口（如"如何申领社保卡"） | P1 |

#### 3.2.4 敏感词过滤需求

| 编号 | 需求项 | 描述 |
|------|--------|------|
| SF-01 | 敏感词库 | 预置政治敏感词、违法违规词、暴力恐怖词等词库 |
| SF-02 | DFA算法 | 采用DFA（确定有限自动机）算法实现高效匹配 |
| SF-03 | 双向过滤 | 输入预过滤（用户提问）+ 输出后过滤（AI回答） |
| SF-04 | 敏感词替换 | 命中敏感词以 `***` 替换，不阻断对话 |
| SF-05 | 严重敏感 | 遇严重敏感内容（如涉政涉暴），直接拒绝处理并记录 |
| SF-06 | 日志审计 | 所有敏感词命中记录审计日志（用户、内容、时间、词类） |
| SF-07 | 词库管理 | B端管理界面支持敏感词库的增删改查和批量导入 |
| SF-08 | 变形词识别 | 支持拼音、谐音、拆字等简单变形词识别 |

#### 3.2.5 会话记忆设计

```java
// 会话记忆存储策略
// Redis结构: chat:memory:{sessionId} → List<ChatMessage>
```

| 会话记录 | AI问答会话 | 会话ID、用户ID、消息列表、创建时间 |
| 敏感词 | 敏感词库 | 词ID、词语、分类、级别、状态 |
| 统计数据 | 统计汇总 | 统计维度、指标名、数值、统计周期 |
| 绩效评分 | 绩效考核 | 周期ID、部门ID、指标ID、得分、总分、排名 |

### 8.2 数据保留策略

| 数据类型 | 保留期限 | 归档策略 |
|----------|----------|----------|
| 业务数据 | 永久 | 在线3年，之后归档冷存储 |
| 审计日志 | ≥6个月 | 在线6月，之后归档 |
| AI会话记录 | 90天 | 90天后自动删除 |
| 操作日志 | 180天 | 180天后归档 |
| 统计数据 | 永久 | 汇总数据永久保留，明细数据保留1年 |
